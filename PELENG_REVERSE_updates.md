# Новые находки по реверсу PelengPC / peleng_clone

Этот файл дополняет `PELENG_REVERSE.md` только теми деталями, которые
проявились при повторном анализе кода `peleng_clone.py` и сверке с
Markdown-документом.

## 1. Импорт FDB идёт в два слоя, а не в один

Из кода видно, что импорт оригинальной базы `PelengPC.fdb` устроен как
многошаговый процесс:

1. читаются строки из таблиц `NASTR*`, `RESULTS*`, `SHORTPROT*`;
2. каждая строка зеркалируется в общую таблицу `Measurements`;
3. отдельным проходом подтягиваются только те `BLOCKZAP`, на которые есть
   ссылки через `NUMKOD`;
4. после этого вызывается `enrich_measurements_from_blocks()`, чтобы
   вытащить поля из TLV/BCD-тела и заполнить пропуски.

Это важно: импорт не просто копирует строки, а строит нормализованное и
обогащённое представление данных.

## 2. BLOCKZAP импортируется выборочно

Код не тащит весь `BLOCKZAP` подряд. Он сначала собирает все нужные `NUMKOD`,
затем читает только соответствующие `NUMBER` из `BLOCKZAP`.

Особенность:
- запросы идут чанками по 500 значений;
- если `BLOCKZAP` для какого-то `NUMKOD` не найден, это считается как
  пропуск, а не авария импорта;
- после загрузки сырых BLOB выполняется второй проход по `Measurements`.

То есть клон рассчитан на частично битые базы и не падает на отдельных
проблемных записях.

## 3. Поля из FDB сохраняются в общей таблице Measurements

`Measurements` используется как единый слой для:
- данных, принятых по UART от УД2-102;
- данных, пришедших из FDB;
- обогащённых полей из `BLOCKZAP`.

Важные поля, которые реально используются для FDB:
- `TimeStr`
- `Operator`
- `IndMaker`
- `MakeTime`
- `Defekt`
- `CodeDef`
- `TypeVar`
- `NumKod`
- `Source`

Это означает, что клон уже умеет хранить не только сырой протокол, но и
операторские/служебные поля оригинальной БД.

## 4. Источник записи помечается явно

Для каждой строки есть поле `Source`.

По коду встречаются значения:
- `UD2`
- `FDB-NASTR`
- `FDB-RESULTS`
- `FDB-SHORTPROT`

Это удобно для отделения живого UART-потока от импортированной базы и для
фильтрации в GUI.

## 5. A-развёртка в GUI уже не просто “есть”, а оформлена как отдельный виджет

Виджет `AScanWidget`:
- рисует сетку;
- использует оси с подписями в процентах и дБ;
- показывает шкалу слева/справа и нижнюю шкалу по времени/глубине;
- умеет помечать зону через `set_zone_label()`.

Отдельная деталь: в коде привязка шкалы сделана так, что
`100 %` соответствует `26 дБ`. Это уже не просто «линия на графике», а
специально нормированная визуализация протокола.

## 6. A-scan привязывается к зоне SHORTPROT2

В `AScanWidget` есть поля:
- `self._zone_label`
- `self._zone_meta`

Комментарий в коде явно указывает, что это нужно для `SHORTPROT2`:
`"Зона N из M"` и дополнительная строка с объектом/кодом дефекта.

Это важный штрих: график A-развёртки не живёт сам по себе, а связан с
отчётной логикой протокола.

## 7. B-развёртка тоже реализована как живой рендер, а не заглушка

`BScanWidget`:
- строит изображение из матрицы пикселей;
- преобразует амплитуду в цвет;
- использует `QImage.Format_ARGB32`;
- имеет цветовую шкалу справа.

То есть B-scan в клоне не “ожидает будущей реализации”, а уже отрисовывается
в GUI.

## 8. В таблице Measurements инициализация уже учитывает старые версии БД

При старте `BlockZapDB` делает миграции:
- добавляет новые колонки, если их нет;
- создаёт индексы по `Source` и `NumKod`;
- допускает старую структуру SQLite и доращивает её.

Это значит, что база клона эволюционирует без полной пересборки.

## 9. Контроль строк в FDB уже есть

В коде есть очистка неотображаемых символов из строк FDB.
Это сделано потому, что в Qt такие символы могут отображаться как
«квадратики».

С точки зрения реверса это полезно: значит, автор клона уже столкнулся с
реальными проблемными строками в исходной базе и закрыл это на уровне импорта.

## 10. Статус покрытия по итогам повторного анализа

### Уже покрыто
- TLV-поток PelengPC
- A-scan визуализация
- B-scan визуализация
- импорт FDB в SQLite
- зеркалирование FDB в `Measurements`
- обогащение из `BLOCKZAP`
- разбор BCD-паспортов УД2-102
- связка `SHORTPROT2` → зона/график

### Всё ещё выглядит неполным
- полный парсер `.dal`
- полная автоматическая генерация полей для всех TLV-схем
- точная валидация на живом `PelengPC.fdb` для проверки всех веток
- 1:1 воспроизведение оригинальных FastReport-шаблонов `.fr3`

## 12. Короткий вывод

По коду клон уже умеет:
- импортировать FDB с обогащением;
- сохранять протоколы в единую таблицу;
- показывать A- и B-развёртку;
- связывать протокол с отчётом и зоной контроля.

Главный оставшийся разрыв — не в просмотре данных, а в полном
автоматическом восстановлении оригинальных схем и отчётных шаблонов.
